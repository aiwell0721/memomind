# MemoMind 项目开发准则

> **本文档由两部分组成**：Karpathy 基础编码准则 + MemoMind 文档结构约束。两者并重，不可违背。

---

## 一、基础编码准则（Karpathy 规则，不可违背）

### 1. 编码前先思考
**不要假设。不要隐藏困惑。暴露权衡。**
- 实施前：明确陈述你的假设。如果不确定，先问我。
- 如果存在多种解释，把它们列出来——不要默默替我做决定。
- 如果有更简单的方法，说出来。必要时敢于反驳我的需求。
- 如果有不清楚的地方，停下来。指出是什么让你困惑，然后提问。

### 2. 极简优先
**用最少的代码解决问题。拒绝一切 speculative（投机性）代码。**
- 不实现未被要求的功能。
- 不为一次性代码做抽象。
- 不做未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行但本来 50 行就能搞定，重写它。
- 问问自己："资深工程师会觉得这过度设计了吗？"如果是，简化它。

### 3. 手术刀式修改
**只碰你必须碰的。只清理你自己弄乱的地方。**
- 编辑现有代码时：
  - 不要"顺手"改进相邻的代码、注释或格式。
  - 不要重构没坏的东西。
  - 匹配现有风格，即使你会用不同的写法。
  - 如果注意到无关的死代码，提一嘴——别直接删。
- 当你的修改产生孤儿代码时：
  - 删除**因为你的修改**而变得无用的 import/变量/函数。
  - 除非我要求，否则不要删除以前就存在的死代码。
- **检验标准**：每一行改动都必须直接追溯到我的请求。

### 4. 目标驱动执行
**定义成功标准。循环验证直到完成。**
- 将任务转化为可验证的目标：
  - "加个验证" → "写无效输入的测试，然后让它们通过"
  - "修复 bug" → "写个能复现它的测试，然后让它们通过"
  - "重构 X" → "确保重构前后测试都能通过"
- 对于多步骤任务，陈述简短计划：
  1. [步骤] → 验证：[检查项]
  2. [步骤] → 验证：[检查项]
  3. [步骤] → 验证：[检查项]
- 强有力的成功标准让你能独立循环；软弱的标准（"让它跑起来"）需要不断向我确认。

## 二、MemoMind 文档结构约束（不可违背）

### 文档分层分类体系：3 层 5 类

所有项目文档存放在 `docs-project/` 目录，按以下结构组织：

```
docs-project/
├── 01-product-concept/       ← 第一层·第1类：产品概念及商业逻辑
├── 02-business-logic/        ← 第一层·第2类：产品业务逻辑
├── 03-architecture/          ← 第二层·第3类：产品架构设计
├── 04-detailed-design/       ← 第二层·第4类：产品详细设计
└── 05-development-plan/      ← 第三层·第5类：产品开发计划
```

### 5 类文档定义

| 层级 | 类别 | 目录 | 内容 |
|------|------|------|------|
| **第一层** | 第1类 | `01-product-concept/` | PRD、竞品分析、商业模型、产品概述、核心功能定义 |
| **第一层** | 第2类 | `02-business-logic/` | 决策日志、风险登记、MVP定义、业务规则、用户故事 |
| **第二层** | 第3类 | `03-architecture/` | 整体架构、模块架构、数据模型、技术选型 |
| **第二层** | 第4类 | `04-detailed-design/` | API规范、接口设计、实现细节、算法设计 |
| **第三层** | 第5类 | `05-development-plan/` | Phase计划、迭代计划、里程碑、风险缓解计划 |

### 文档写入规则

1. **分类不可混淆**：
   - 产品概念类文档 → 只能放在 `01-product-concept/`
   - 业务逻辑类文档 → 只能放在 `02-business-logic/`
   - 架构设计类文档 → 只能放在 `03-architecture/`
   - 详细设计类文档 → 只能放在 `04-detailed-design/`
   - 开发计划类文档 → 只能放在 `05-development-plan/`

2. **命名规范**：`NN-文档标题.md`（两位数字序号 + 中文标题）

3. **文档先行**：任何需求变更、设计决策、功能新增，必须先更新对应类别的文档，再改代码。代码不得领先文档。

4. **索引维护**：新增/修改文档后，必须同步更新 `docs-project/README.md` 索引。

5. **发布文档**：与产品发布相关的文档（部署指南、用户手册、API参考、快速入门、故障排查）存放在 `docs/` 目录，不归入 3 层 5 类体系。

### 判断文档归属的快速指南

| 如果你要写... | 放到... |
|--------------|---------|
| "这个产品是什么、解决什么问题" | 01-product-concept |
| "用户怎么用、业务流程是什么" | 02-business-logic |
| "系统由哪些模块组成、数据怎么流转" | 03-architecture |
| "这个API的参数列表、这个函数的实现逻辑" | 04-detailed-design |
| "下个Phase做什么、什么时候交付" | 05-development-plan |
| "怎么安装部署、怎么用" | docs/（发布文档） |

## 三、开发习惯与规范

### 语言
- 始终使用**中文**交流、解释思路和写注释。
- 代码变量/函数/类名使用**英文**（驼峰或下划线），注释用中文。

### Python
- 类型提示是必须的（函数参数和返回值）。
- 文档字符串使用 Google 风格。

### Git
- Commit Message 使用中文，格式：`<类型>: <中文描述>`
- 示例：`修复: 解决用户登录时 token 过期未刷新的问题`
- PR 审查时文档变更和代码变更应在同一次提交中。

### 前端
- 默认使用 Tailwind CSS，不主动引入 UI 组件库。
