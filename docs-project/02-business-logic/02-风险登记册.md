## 高风险 (High)

### RISK-001: MVP范围膨胀导致延期

| 属性 | 内容 |
|------|------|
| **风险ID** | RISK-001 |
| **风险描述** | MVP功能过多，开发团队无法在计划时间内完成 |
| **风险类别** | 项目/进度 |
| **可能性** | 高 (70%) |
| **影响程度** | 高 (延期3个月以上) |
| **风险等级** | 🔴 极高 |
| **负责人** | 项目经理 |
| **状态** | 活跃 |

**风险详情**:
- 原始PRD包含50+功能，计划8周完成
- 实际需要16-20周才能高质量交付
- 团队可能被压垮，导致质量下降

**应对策略**:
- ✅ **规避**: 大幅削减MVP范围至核心5项功能
- ✅ **缓解**: 采用16周开发周期，预留4周缓冲
- ⬜ **转移**: 考虑外包非核心功能
- ⬜ **接受**: 接受部分功能延后

**行动计划**:
1. [ ] 重新评审PRD v2，确认功能范围
2. [ ] 与开发团队确认工时估算
3. [ ] 建立每周进度检查机制
4. [ ] 准备功能裁剪预案

**监控指标**:
- 每周完成的功能点数量
- 实际工时 vs 估算工时偏差
- Bug数量和严重程度

## 附录

### A. 风险登记流程

1. **识别**: 通过头脑风暴、历史项目、专家访谈识别风险
2. **评估**: 评估可能性和影响程度
3. **应对**: 制定规避、缓解、转移或接受策略
4. **监控**: 定期审查风险状态
5. **报告**: 向相关方报告高风险项

### B. 风险等级定义

| 等级 | 可能性 | 影响程度 |
|------|--------|----------|
| 🔴 高 | >50% | 项目失败/重大损失 |
| 🟡 中 | 30-50% | 显著延期/成本增加 |
| 🟢 低 | <30% | 轻微影响 |
