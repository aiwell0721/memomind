## OPT-01：FTS5 同步从「整表重建」改为「增量更新」

### 问题

`core/database.py:_sync_fts_if_notes_write` 当前实现里，notes 主表的每次 UPDATE/DELETE 都触发 `_fts_rebuild_all()`——重建整张 `notes_fts` 表。

### 原因

Python `sqlite3` 模块不返回 UPDATE/DELETE 命中行的 id 列表，本轮代码（commit `e152b00`）只能折中。已在 DEC-011 标注遗留。

### 触发条件（量化）

当 `notes` 行数 ≥ **2000** 时，每次 UPDATE 会变慢到肉眼可感（jieba 分词每条 1–3ms，2000 条全表重建 ≈ 3–6 秒，会让 API 请求超时）。

### 实施方案

**方案一（轻量，0.5 天）**：SQL 正则解析

1. 在 `Database.execute()` 里，对匹配 `_RE_UPDATE_NOTES` / `_RE_DELETE_NOTES` 的 SQL，**执行前**先 `SELECT id FROM notes WHERE ...` 查出受影响 id，记到 `_fts_pending_ids: set[int]`。
2. 执行原 SQL。
3. 执行后从 `_fts_pending_ids` 逐一刷新（DELETE 走 `_fts_delete`，UPDATE 走 `_fts_upsert`）。

```python
# 伪代码骨架（不实施）
_RE_UPDATE_WHERE = re.compile(r"UPDATE\s+notes\b(?!_)\s+SET\s+.*?(\sWHERE\s+.+)?$", re.I|re.S)
_RE_DELETE_WHERE = re.compile(r"DELETE\s+FROM\s+notes\b(?!_)(\s+WHERE\s+.+)?$", re.I|re.S)

def execute(self, sql, params=()):
    op, where_clause, where_params = self._classify_notes_write(sql, params)
    affected_ids = []
    if op in ('update', 'delete'):
        affected_ids = [r[0] for r in self.conn.execute(
            f"SELECT id FROM notes {where_clause or ''}", where_params
        ).fetchall()]
    cursor = self.conn.cursor()
    cursor.execute(sql, params)
    if op == 'insert':
        self._fts_upsert_by_id(cursor.lastrowid)
    elif op == 'update':
        for nid in affected_ids:
            self._fts_upsert_by_id(nid)
    elif op == 'delete':
        for nid in affected_ids:
            self._fts_delete(nid)
    return cursor
```

**坑**：

- **UPDATE 只改 `workspace_id`**——FTS5 不索引 workspace_id，理论上可零开销跳过同步。子优化 OPT-01.a：识别 UPDATE SET 列表是否含 title/content/tags，否则跳过。
- **params 切片**：`UPDATE notes SET title = ? WHERE id = ?` 共两个 `?`，预查 SQL 是 `SELECT id FROM notes WHERE id = ?`，必须取 `params[-1:]`。需要解析 SET 段的 `?` 数量——是方案一最脆的环节。

**方案二（彻底，2 天）**：显式 API

跳过 SQL 解析，封装 `update_notes(where_sql, where_params, set_dict)` 显式 API，逐步迁移现有 11 处 `db.execute("UPDATE notes ...")`。更稳但工作量大。

### 风险

方案一中等（SQL 解析脆弱）、方案二低（显式 API）。

### 推荐时机

notes 表行数监控 ≥ 1000 时启动；或下次性能基准（`benchmarks/`）出现回归时启动。**当前不做** —— 典型 speculative，应先有真实痛点。

## OPT-02：搜索语义 OR → AND（可配置）

### 问题

`core/tokenizer.py:111` `tokenize_for_search` 用 `' OR '.join()` 把分词后 token 用 OR 连。用户搜 "全文搜索"，jieba 分成 `["全文", "搜索"]`，FTS5 收到 `全文 OR 搜索`——任意一个词命中都返回，召回大、精度低。

### 真实痛点

笔记越多，OR 语义返回的"噪声"越多；BM25 排序能部分缓解，但用户体验是"搜索质量越用越差"。

### 实施方案

1. `tokenize_for_search(query, mode: Literal['or', 'and'] = 'or')`。
2. `mode='and'` 时，分词结果用空格连接（FTS5 默认 AND）。
3. `SearchService.search()` 新增 `match_mode` 参数。
4. REST `/api/notes/search` 加 query param `mode=and|or`，默认 `or`（向后兼容）。
5. **更聪明的策略**："多词查询用 AND，单词查询用 OR"——但这是产品决策，需用户确认。

### RED 测试

构造笔记含 "测试"，不含 "全文"。AND 模式搜 "全文测试" 应返回 0 条；OR 模式应返回 1 条。

### 风险

低。纯参数加法，不破坏现有行为。

### 推荐时机

与 OPT-04 一起做"搜索体验改进"一波；或用户反馈"无关结果太多"时立刻做。

## OPT-03：WAL 文件清理 - 写入部署文档

### 问题

部署报告问题 4 描述：删 `memomind.db` 但未删 `memomind.db-wal` / `memomind.db-shm`，SQLite 自动应用残留 WAL，产生"幽灵数据"。这是**部署/运维操作**问题，不是代码 bug。

### 实施方案

1. 在 `docs/` 目录（按 CLAUDE.md：产品文档，面向运维）找现有部署文档，追加"重置数据库"小节。无则新建 `docs/02-部署运维/01-数据库管理.md`。
2. 内容（约 30 行）：
   - WAL 模式三文件作用图（`.db`、`.db-wal`、`.db-shm`）
   - 重置数据库正确流程（关进程 → 删三文件 → 重启）
   - 在线替代方案：`PRAGMA wal_checkpoint(TRUNCATE); VACUUM;`
   - 备份时也要包含三文件（或先 checkpoint 再只备份主文件）
3. **代码层补强（可选）**：在 `core/backup_service.py` 备份前自动跑 `PRAGMA wal_checkpoint(TRUNCATE)`——同时修复"备份恢复后丢最近 N 分钟写入"的隐藏 bug。

### 风险

零。

### 推荐时机

随时。优先级低但成本极低。

## OPT-04：停用词写入/查询对称化

### 问题

`core/tokenizer.py` 的 `tokenize()` 默认 `remove_stopwords=True`，停用词表里有 "如何"、"什么"、"为什么"。但 `_tokenize_for_fts()` 写入索引时设置 `remove_stopwords=False`，`tokenize_for_search` 查询时走 `remove_stopwords=True`——**写入/查询不对称**。

### 症状

- 用户笔记标题 "如何做笔记" 被索引为 `[如何, 做, 笔记]`，搜索 "如何做笔记" 被分成 `[做, 笔记]`（停用词 "如何" 被移除）。仍能命中（子集匹配），但 BM25 分数不准。
- 用户搜 "如何"，分词后是空集，返回 0 结果——但索引里其实有这个词。

### 真实风险

中等。停用词移除对中文搜索的影响实际**比英文大**——"如何" 在英文里=how，但对中文搜索它常是关键词。

### 实施方案

三个选项：

- **A（推荐）**：写入和查询都 `remove_stopwords=False`。停用词留给 BM25 自然降权。
- B：写入和查询都 `remove_stopwords=True`（一致即可）。
- C：精简停用词表，只移除真正无信号的 "的/了/和/是"，去掉 "如何/什么/为什么"。

### RED 测试

用户搜 "如何做笔记" 应返回标题为 "如何做笔记" 的那条。

### 风险

低，但需要重新 reindex（启动时已自动跑）。

### 推荐时机

下次。真问题但不阻塞。

## OPT-05：jieba 用户词典 - 项目术语对齐

### 问题

`core/tokenizer.py:25-37` 已有自定义词典（"人工智能"、"MemoMind" 等 18 个词），但写死在代码里。若团队知识库高频出现 "RAG"、"GraphRAG"、"LLM 评测"、"prompt engineering" 等术语，jieba 默认分词会切错。

### 实施方案

1. 支持从 `~/.memomind/user_dict.txt` 加载用户词典（一行一个词，可选词频）。
2. `ChineseTokenizer.__init__` 末尾 `jieba.load_userdict(path)`，文件不存在则跳过。
3. CLI：
   - `memomind dict add <词>`
   - `memomind dict list`
   - `memomind dict remove <词>`
4. 加词后需 `reindex_notes()`，CLI 提示用户运行 `memomind reindex`。

### 风险

低。

### 推荐时机

当用户反馈 "我们公司的术语 X 搜不准" 时。在那之前是 speculative。

## OPT-06：API 启动日志改用 logging（替换 print）

### 问题

`core/api_server.py` 用 `print()` 输出启动信息（reindex 提示等）。本轮**有意没换** logging（法则三：匹配现有风格）。

### 真实成本

生产环境用 systemd/Docker 启动时，print 输出到 stdout 仍能被收集，但**没有日志级别、时间戳、模块名**。日志聚合（Loki / ELK）按级别过滤就废了。

### 实施方案

1. 新建 `core/logging_config.py`：根据环境变量 `MEMOMIND_LOG_LEVEL`（默认 INFO）配置 root logger。
2. 各模块 `logger = logging.getLogger(__name__)`。
3. `grep -rn "print(" core/` 找出现有 print，逐个替换为 `logger.info/warning`。
4. 测试可用 `caplog` fixture 捕获。

### 风险

低，但属于全项目改动，diff 较大，需单独 PR。

### 推荐时机

生产部署前。当前纯本地开发可推迟。

## OPT-07：管理员创建用户 - 一次性临时密码 + 强制改密

### 问题

本轮 Bug 4 修复让管理员手动输入密码，但这创造了**新的安全反模式**——管理员知道用户的密码（哪怕是初始的）。OWASP 标准做法是 "一次性密码 / 强制首次登录改密"。

### 实施方案

1. 后端：`POST /api/users` 不要求 password，自动生成 16 字符 base32 临时密码 + `force_password_change=True` 字段。
2. 后端：返回临时密码 + 提示"请通过安全渠道发送给用户，首次登录后必须改密"。
3. 后端：用户表加 `must_change_password BOOLEAN DEFAULT FALSE`。
4. 后端：登录后若 `must_change_password=True`，返回特殊 token 只允许调 `/api/users/me/password`。
5. 前端：Settings 创建用户成功后弹 modal 显示临时密码 + 复制按钮。
6. 前端：登录成功若服务端要求改密，强制跳改密页面。

### 风险

中。涉及登录流程改造。

### 推荐时机

和 refresh token 一起做。MVP 阶段可暂缓——团队知识库不是 SaaS，攻击面有限。

零。

随时。是法则六的精神延伸——经验沉淀。
