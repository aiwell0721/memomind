## 1. 背景与目标

### 1.1 业务背景

知识工作者面临信息过载和检索困难的问题：
- 笔记分散在不同工具，难以统一管理
- 传统搜索依赖关键词，无法语义理解
- AI助手缺乏对个人信息的知识储备
- 现有工具要么太简单（Apple Notes），要么太复杂（Obsidian）

### 1.2 用户痛点

| 用户类型 | 核心痛点 | 现有方案缺陷 |
|----------|----------|--------------|
| 开发者 | 代码片段、技术文档分散 | Cursor只管代码，Notion太笨重 |
| 技术写作者 | 难以组织研究素材 | 传统笔记缺乏AI能力 |
| 产品经理 | 需求文档、用户反馈分散 | 搜索困难，难以关联 |

### 1.3 产品定位

**新定位**: "为技术工作者打造的AI原生笔记工具"

**核心价值**:
1. **AI理解你的笔记** - 语义搜索，自然语言提问
2. **数据完全掌控** - 本地存储，开放格式
3. **开发者友好** - 简洁API，易于集成

**目标用户优先级**:
1. **P0**: 开发者和技术写作者（早期采用者）
2. **P1**: 产品经理和研究员
3. **P2**: 普通知识工作者

## 2. 用户故事

### 2.1 开发者用户

**US-001: 代码知识管理**
> 作为一名开发者，我希望将代码片段、技术文档和学习笔记统一管理，AI能够理解我的技术栈并提供相关建议。

**US-002: 快速检索**
> 作为一名开发者，我希望用自然语言搜索笔记（如"查找关于Redis集群配置的笔记"），而不是记住具体关键词。

### 2.2 技术写作者

**US-003: 研究素材整理**
> 作为一名技术写作者，我希望AI帮我总结和关联研究素材，以便快速撰写文章。

**US-004: 写作辅助**
> 作为一名写作者，我希望AI能基于我的笔记库回答相关问题，提供写作灵感。

## 4. 技术架构 (简化版)

### 4.1 技术栈选型

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| **桌面端** | Electron | 成熟稳定，生态丰富 |
| **前端** | React + TypeScript | 团队熟悉，效率高 |
| **状态管理** | Zustand | 轻量高效 |
| **编辑器** | CodeMirror 6 | 功能强大，可扩展 |
| **存储** | SQLite (单一数据库) | 简单可靠 |
| **向量检索** | SQLite + 简单向量扩展 | 避免引入新数据库 |
| **AI** | OpenAI API | 成熟可靠，效果最佳 |
| **搜索** | SQLite FTS + 向量相似度 | 简化架构 |

### 4.2 架构图

```
┌─────────────────────────────────────────┐
│              用户界面层                   │
│  ┌─────────────┐ ┌─────────────────┐   │
│  │   笔记编辑器  │ │   搜索/AI面板    │   │
│  └─────────────┘ └─────────────────┘   │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│              应用核心层                   │
│  ┌─────────────┐ ┌─────────────────┐   │
│  │   笔记管理    │ │   AI服务        │   │
│  └─────────────┘ └─────────────────┘   │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│              数据存储层                   │
│  ┌─────────────┐ ┌─────────────────┐   │
│  │   SQLite    │ │   本地文件系统    │   │
│  │ (元数据+索引)│ │   (Markdown文件)│   │
│  └─────────────┘ └─────────────────┘   │
└─────────────────────────────────────────┘
```

### 4.3 数据模型 (简化)

```typescript
interface Note {
  id: string;              // UUID
  title: string;           // 标题
  content: string;         // Markdown内容
  folderPath: string;      // 文件夹路径
  tags: string[];          // 标签
  createdAt: Date;         // 创建时间
  updatedAt: Date;         // 更新时间
  embedding?: number[];    // 向量表示（可选）
}
```

## 6. 商业模式

### 6.1 定价策略

**Free (免费)**:
- 本地使用
- 基础笔记功能
- 自带OpenAI API Key使用AI功能

**Pro ($8/月 或 $80/年)**:
- 所有免费功能
- 云同步（Phase 2）
- 内置AI额度（每月1000次调用）
- 优先支持

### 6.2 成本估算

| 成本项 | 月成本 | 说明 |
|--------|--------|------|
| OpenAI API | $2-5/用户 | 按使用量计费 |
| 云存储 | $0.5/用户 | 假设平均1GB |
| 基础设施 | $500 | 固定成本 |

**盈亏平衡点**: 约500个Pro用户

## 7. 竞品差异化

### 7.1 与Obsidian对比

| 维度 | Obsidian | MemoMind |
|------|----------|----------|
| AI能力 | 依赖插件 | 原生集成 |
| 易用性 | 学习曲线陡峭 | 开箱即用 |
| 开发者友好 | 插件开发复杂 | 简洁API |
| 数据格式 | Markdown | Markdown |

**差异化**: AI原生 + 更简洁的体验

### 7.2 与Notion对比

| 维度 | Notion | MemoMind |
|------|--------|----------|
| 数据主权 | 云端锁定 | 本地优先 |
| AI能力 | 基础 | 深度集成 |
| 性能 | 大文档慢 | 本地快速 |
| 离线能力 | 有限 | 完全离线 |

**差异化**: 本地优先 + 数据主权

### 7.3 与Mem对比

| 维度 | Mem | MemoMind |
|------|-----|----------|
| 数据存储 | 云端 | 本地 |
| 价格 | $8-15/月 | 免费起步 |
| 开发者API | 有限 | 开放 |
| 隐私 | 平台控制 | 用户控制 |

**差异化**: 免费起步 + 数据主权 + 开发者友好
